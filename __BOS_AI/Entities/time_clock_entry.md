# BOS Entity — time_clock_entry

Entity Type ID: `time_clock_entry`
Bundle: `entry` (single bundle)
Storage: ECK
Activated: 2026-04-26 (Phase 1A)

---

## Purpose

The home for **compensable hours captured by external time clock systems
and paper-time-card entry**. Initial source: Timetrax. Designed to extend
to additional or replacement systems without renaming fields.

This is **distinct** from `wo_time_clock`. The two coexist deliberately:

| Entity            | Purpose                                                         | Drives        |
| ----------------- | --------------------------------------------------------------- | ------------- |
| `wo_time_clock`   | Crew time spent on a specific Work Order.                       | WO billing.   |
| `time_clock_entry`| All other compensable time (shop, training, drive, admin) plus retroactive paper-card entry. | Payroll only. |

`time_clock_entry` rows are **never read for WO billing**. WO labor cost
is computed exclusively from `wo_time_clock`. This separation is a
non-negotiable invariant — see "Non-Negotiables" below.

| Value                | Label                |
| -------------------- | -------------------- |

| `timetrax_import`    | Timetrax Import      |

The list is intended to extend over time. When BOS adopts an additional

ones for audit purposes.

## Mapping to External Clock Systems (User-Side)

External clock systems use their own employee-id namespaces. To attribute
imported rows to the correct BOS user, two new fields live on the
**user** entity (added Phase 1A):

| User field                  | Type        | Purpose                                        |
| --------------------------- | ----------- | ---------------------------------------------- |
| `field_time_clock_id`       | string(50)  | The employee ID assigned to this user by the external system. Stored as a string so leading zeros are preserved. |
| `field_time_clock_system`   | list_string | Which external system the ID corresponds to (e.g., `timetrax`). |

Both live in an "External Time Clock Mapping" fieldset on the user
edit form. The fieldset is **only rendered when the user being edited
has the `teammates` role** — non-teammates (clients, applicants,
public users) never see it. New users (no roles yet) also have it
hidden until first save with the `teammates` role assigned. Gating is
implemented by the [`bos_user_time_clock_mapping`](#related-modules)
module.

### Critical Invariant: `field_time_clock_id` MUST Be Unique

Duplicate IDs across users would cause imported time records to attribute
incorrectly. **This constraint is documented but not yet enforced in
code.** Future import modules MUST validate uniqueness before importing,
and any future admin UI that allows editing the field SHOULD warn on
collision. A custom validator on the user entity is a reasonable
follow-up but is out of scope for Phase 1A.

When `field_time_clock_id` is populated, `field_time_clock_system`
should also be populated. Phase 1A does not enforce this in config —
it is a documented expectation that import modules and admin UI must
respect.

## Idempotency Pattern

`field_external_punch_id` exists so that **the same punch event from an
external system can be processed more than once without creating
duplicate rows**. External clock systems (TimeTrax to start, others
later) can be re-polled, replayed from a backup, or restarted after a
network blip — each event must produce **exactly one** `time_clock_entry`
row regardless of how many times it is seen.

### How it is enforced

* **Import modules MUST set `field_external_punch_id`** to the external
  system's primary key for the punch (e.g., the TimeTrax `PunchKey`
  column).
* **Before creating a new row, the import module MUST query for an
  existing row with the same `field_external_punch_id`** and skip
  insertion if one is found. This is the only enforcement point.
* Drupal field config does **not** enforce uniqueness on this field. A
  custom validator was rejected as premature — the import module is
  the only writer of this field, so the check belongs there.
* For manual entries created via the canonical add URL,
  `field_external_punch_id` stays NULL. The form display hides the
  field on add/edit so humans never see or set it. NULL values do not
  participate in the uniqueness contract — many manually-entered rows
  can coexist with NULL.

### Why this design over alternatives

* **Per-event storage (one event = one row)** — the existing
  `field_clocking_in_out` (In / Out / Manually Entered) captures
  direction; pairing into shifts happens in Drupal Views, not at
  import time. So the idempotency key is per-event, not per-shift.
* **No paired-shift `start_time`/`end_time` fields** — would force
  pairing logic at import time and complicate the idempotency contract
  (what if only one half of a pair has been imported?).
* **No total-hours field on the row** — Views do the math on demand.

## Non-Negotiables

1. **Never used for WO billing.** WO labor cost is computed only from
   `wo_time_clock`. Reading `time_clock_entry` rows during WO billing
   recalc is forbidden.
2. **Source is mandatory.** Every row has a `field_source`. The default
   is `live_punch`; import modules MUST explicitly set the appropriate
   value (`timetrax_import`, etc.).
3. **No deletion.** Payroll record. Office Managers may correct values
   on a row but should not delete one.
4. **External IDs preserved as strings.** `field_time_clock_id` is a
   string (max 50) — never an integer — so leading zeros and
   alphanumeric IDs survive import.
5. **`live_punch` is meant to be permanent on a row.** Once a row is
   captured as a live punch, it should not later be re-labeled as
   `manually_entered` or `timetrax_import`. Future hardening should
   make `field_source` read-only on existing rows where the current
   value is `live_punch`, for non-admin roles. Phase 1A does not
   implement this gate — it is documented as expected behavior for the
   manual-entry UI phase.

## Permissions

Granted in config to all four operational roles
(`teammates`, `supervisor`, `administration`, `site_admin`) — pre-existed
from when the entity was first scaffolded. Phase 1A did not modify
permissions. See `user.role.*.yml` in `config/sync/` for the full grants.

`administrator` has full access via the is_admin flag.

## Where Rows Come From

**Phase 1A:** Foundation only. No code path creates rows yet. The only
way to add a row is the canonical ECK add URL (`/time_clock_entry/add`),
where it will default `field_source` to `live_punch`.

**Phase 1B (planned):** Timetrax CSV import module — will set
`field_source = timetrax_import`, `field_team_member` resolved via
`field_time_clock_id` lookup on the user entity.

**Future phases:** Admin UI for supervisor manual entry (sets
`field_source = manually_entered`); system corrections (sets
`field_source = system_correction`).

Clock Mapping fieldset on the user edit form, via the new
`bos_user_time_clock_mapping` module.

**Phase 1B (2026-04-26).** Idempotency support:

* `field_external_punch_id` added (string max 100, optional, hidden on
  the form display, shown in default view display for debugging).
* Documented [Idempotency Pattern](#idempotency-pattern) — uniqueness
  enforced by import modules at write time, not by Drupal field
  config; manual entries leave the field NULL.

Out of scope and explicitly NOT yet implemented:

* No Timetrax import module (Phase 1D).
* No admin UI for supervisor manual entry.
* No PHP enforcement of the `field_time_clock_id` uniqueness invariant.
* No PHP enforcement of the `field_external_punch_id` uniqueness
  invariant — see [Idempotency Pattern](#idempotency-pattern).
* No PHP enforcement of `field_source` read-only-after-live-punch.
* No changes to `wo_time_clock` or any `wo_*` modules.
